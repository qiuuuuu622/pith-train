# 工作日志 2026-06-16：35B MoE 长序列训练优化

目标：在单台 8×H100 节点上训练 Qwen3.5-35B-A3B（GDN 混合，256 专家 top-8），
训练效率超过 Megatron。本轮聚焦目标拆解的前两项优化：**层级激活重计算** 与
（下一步）**异步重叠优化器步**。

## 基线与并行配置

- 模型：Qwen3.5-35B-A3B 文本侧，hidden 2048、40 层（30 GatedDeltaNet 线性注意力 +
  10 全注意力，`full_attention_interval=4`），vocab 248320，256 专家 top-8 + 共享专家。
- 并行：`PP=4 × EP=2`，world=8 全部用尽 → DP=1。专家优化器状态（fp32 动量约
  48GB/rank）必须走 FSDP2 `CPUOffloadPolicy`。无 TP。
- DualPipeV 双向流水 + 零气泡 deferred wgrad（`WeightGradStore`）+ EP all-to-all +
  CP zigzag ring attention。

## 1. Triton 交叉熵梯度 bug 修复

`operators/cross_entropy.py` 原 triton kernel 的 loss 正确但 **梯度错误**
（与 `F.cross_entropy` 余弦相似度仅 0.076、量级约 51×，且存在并发写竞争）。
已整体替换为纯 torch 的分块在线 softmax 实现（按 token 分块，原地写入梯度），
数值与 `F.cross_entropy` 对齐。新增 `tests/test_cross_entropy.py` 覆盖。

## 2. Fused Linear Cross-Entropy（FLCE）

新增 `operators/fused_linear_cross_entropy.py`：不再 materialize `[N, V]` logits，
在 backward 内重算，`grad_weight` 通过原地 `addmm_` 在 fp32 累加，避免 `[V, H]` 临时张量。

- 接入：`models/qwen3_5_moe.py` 新增 `Qwen3_5MoeLMHead`（`forward→logits`，
  `fused_loss→FLCE`），epilog 融合分支读取 `self._labels`；DualPipeV 两条 compute
  路径打通 `_labels`；`modules/training.py` 对 lm_head 注册
  `register_fsdp_forward_method(lm_head, "fused_loss")`、
  `reshard_after_forward = not fuse_lmhead_ce`。
- 验证：FLCE 输出与 `F.cross_entropy` 余弦 1.0。`tests/test_fused_linear_cross_entropy.py`。
- 收益：lm_head 所在 PP stage 峰值降约 2GB；优化 backward 后消除了之前
  "fused 反而加重 PP stage 间差距" 的回退。

## 3. 层级激活重计算

`layers/factory.py::ModelImplMode` 新增开关 `recompute_mlp`、`recompute_attn`。
关键约束：必须用 `torch.utils.checkpoint(use_reentrant=True)`——`use_reentrant=False`
会与 DualPipe 底层 `Variable._execution_engine.run_backward` + deferred wgrad 路径
**静默冲突**（表现为专家 backward 被跳过：grad-norm 20.6→16.4、吞吐虚高 4×）。
该 bug 靠 grad-norm 校验发现，单看 loss 无法察觉。

- `recompute_mlp`：重算专家 MLP 前向。`dualpipe/execution.py` 新增
  `maybe_recompute_mlp(...)`，在 stage3_f 包裹 `forward_mlp`，开启时跳过
  `gathered_tokens` 提前释放；`dualpipe/modeling.py` 同步处理释放时机。
  实测 rank2-7 显存 nvidia-smi 58→42GB，约省 13GB。
- `recompute_attn`：重算注意力/GDN 块（`_forward_attn_compute`），这是长序列下
  主导的激活（GDN scan + 全注意力）。router/`gate` 留在 checkpoint 外，避免
  load-balance aux loss 在重算时被重复计入——`_forward_attn_compute` 正好返回
  2 个张量，是干净的切分点。

### 已验证

- `RECOMPUTE_MLP=1` @ seq2048：loss 1.001、grad-norm 20.69（与无重计算一致）、
  约 617 tok/s/gpu、约 48GB/rank，正确性与性能均 OK。

### 待续

- seq8192 下 rank0/1（PP 边界 stage，承载 embedding + lm_head + FLCE + 额外激活）
  即便叠加 `recompute_attn` 仍 OOM（PyTorch 已分配约 73GB）。rank2-7 在
  61-65GB 可跑。长序列下边界 stage 的显存收敛是下一步要点。

## 4. 优化器与训练循环

（已在 commit `7f460ef`）`modules/optim.py` 提供 `build_param_groups`、
`scale_and_clip_grad_norm_`；`modules/training.py` 用 AdamW + 参数分组；
`tasks/pretrain_lm.py` 接入梯度缩放/裁剪、GC 节流、`MEM_DEBUG`/`MEMPROF` 诊断。

## 5. 对比 Megatron（基线）

- 相同 seq2048/gbs128：pith-train 约 655 vs Megatron 607 tok/s/gpu（约 +8%）。
- Megatron 最优 @ seq8192/gbs64：约 25 TFLOP/s、约 1311 tok/s/gpu、MFU 约 2.5%。
- 两者在该配置下均受 CPU offload 限制，MFU 偏低。pith-train 的核心差异化在
  DualPipe 调度（含通信/计算重叠）；重计算与异步优化器步属于追平项。
